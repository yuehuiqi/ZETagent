## 🔴 问题一：COMPLETE 泛滥（占失败总数64%）

**现象**：模型在还未完成任务时就输出 `COMPLETE:[]`

**根本原因**：当前 `output_parser.py` 的解析优先级有缺陷：
```python
# 当前代码（错误优先级）：COMPLETE 检测放在最高优先级！
result = _try_parse_complete(raw)
if result:
    return result  # 任何包含"完成"字眼的输出都触发COMPLETE
```

**触发场景**：
- 模型 Thought 中写了"完成步骤1：打开App" → 被误识别为COMPLETE
- 模型输出中文"已完成初始化" → 误触发
- 解析其他动作失败时兜底返回COMPLETE → 大量误触发

**修复方案**：
1. 降低COMPLETE优先级，改为仅在动作部分精确匹配
2. 修改兜底策略：失败时返回CLICK中心点而非COMPLETE

---

## 🟡 问题二：CLICK 坐标偏移（16次，占17%）

**典型错误**：
```
(908, 75)  not in ([27, 977], [104, 158])   # y轴偏上，期望y在104-158，实际75
(500, 170) not in ([793, 964], [53, 115])   # x轴完全偏离，期望x在793-964
(431, 460) not in ([64, 633], [364, 416])   # y轴偏低，期望y在364-416，实际460
```

**根本原因**：
- 模型点击了元素边缘或相邻元素（x范围正确但y偏差）
- 模型对屏幕区域理解偏差（右侧按钮被点到了屏幕中央）
- 图像压缩到720px后，某些小元素的位置感知略有偏差

**修复方案**：
- Prompt中加强"点击元素中心"的约束
- 为常见UI模式（顶部搜索栏、右侧按钮）给出精确坐标范围提示

## 🟡 问题三：OPEN 动作 app_name 乱码（5次）

**现象**：
```
expect 'â£TV', got 'Ӧ�'          # 芒果TV
expect 'ȥ¶¼', got 'Ӧ�'          # 去哪儿
expect 'ϲ¼', got 'Ӧ�'           # 喜马拉雅
```

**根本原因**：模型输出了正确的中文App名，但解析器提取时出现乱码（编码问题），
导致 app_name 被提取为 `Ӧ`（即乱码的"应用"两字）。

**修复方案**：
- 修复 output_parser.py 中的 OPEN 正则提取逻辑
- 在Prompt中要求模型输出精确的App名（与OPEN命令格式一致）
